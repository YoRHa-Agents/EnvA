# Vault File Format Specification

> **摘要 (Chinese Summary):** 本文档定义了 Enva secrets vault 的磁盘文件格式——一个基于别名的密钥池和逐值 AES-256-GCM 加密的单一 JSON 文件。密钥通过人类可读别名标识并映射到环境变量名；应用通过别名引用密钥并可覆盖注入名。密码密钥派生使用 Argon2id (RFC 9106)。加密值采用 SOPS 风格的 `ENC[AES256_GCM,data:…,iv:…,tag:…,type:…]` 编码。文件级完整性通过 HMAC-SHA256 保护（覆盖所有别名、key、value、应用绑定及元数据的规范序列化）。格式自描述：KDF 参数、salt 和版本号嵌入 `_meta`。版本演进遵循语义化版本，minor 版本严格保证向前兼容。
>
> **Update (2026-03-30):** Runtime format `2.1` adds immutable `id` fields to every secret and application record. Top-level maps are still keyed by alias and app name for UX compatibility, but application secret bindings and override maps are normalized to secret ids on save so alias renames remain safe. Existing `2.0` vaults continue to load and are upgraded on save. The current loader-enforced minimum `kdf.memory_cost` is `8192` KiB.

## 1. Vault JSON Schema

A vault file is a single UTF-8-encoded JSON file. The top level contains three fields: `_meta` (metadata), `secrets` (alias-keyed secrets pool with immutable record ids), and `apps` (application reference configuration keyed by app name with immutable app ids).

### 1.1 Top-Level Structure

```json
{
  "_meta": { … },
  "secrets": { … },
  "apps": { … }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `_meta` | `object` | Yes | File metadata including encryption parameters, integrity checksum, and timestamps |
| `secrets` | `object` | Yes | Alias-keyed secrets pool; keys are human-readable aliases, values include an immutable `id`, the env var key, encrypted value, and metadata |
| `apps` | `object` | Yes | Application configuration keyed by app name; each app carries an immutable `id` and stores secret bindings by secret id |

### 1.2 The `_meta` Object

```json
{
  "_meta": {
    "format_version": "2.1",
    "kdf": {
      "algorithm": "argon2id",
      "memory_cost": 65536,
      "time_cost": 3,
      "parallelism": 4
    },
    "salt": "BASE64_ENCODED_32_BYTES",
    "hmac": "BASE64_ENCODED_32_BYTES",
    "created_at": "2026-03-27T10:30:00+08:00",
    "updated_at": "2026-03-27T14:15:30+08:00",
    "created_by": "developer@workstation"
  }
}
```

#### Field Definitions

| Field Path | Type | Required | Description |
|------------|------|----------|-------------|
| `format_version` | `string` | Yes | Format version number following `major.minor` semver. Current version: `"2.1"` |
| `kdf` | `object` | Yes | Key derivation function parameters (self-describing; no external configuration needed for decryption) |
| `kdf.algorithm` | `string` | Yes | KDF algorithm identifier. v2.x only permits `"argon2id"` |
| `kdf.memory_cost` | `integer` | Yes | Argon2id memory cost in KiB. Recommended: `65536` (64 MiB) |
| `kdf.time_cost` | `integer` | Yes | Argon2id iteration count. Recommended: `3` |
| `kdf.parallelism` | `integer` | Yes | Argon2id parallelism (thread count). Recommended: `4` |
| `salt` | `string` | Yes | 32-byte random salt, standard Base64-encoded (with padding). Generated by `os.urandom(32)` |
| `hmac` | `string` | Yes | File integrity checksum — Base64-encoded HMAC-SHA256. See [§4](#4-hmac-integrity) |
| `created_at` | `string` | Yes | Vault creation time in ISO 8601 format with timezone offset |
| `updated_at` | `string` | Yes | Last modification time in ISO 8601 format with timezone offset |
| `created_by` | `string` | Yes | Creator identifier in `user@hostname` format, produced by `getpass.getuser()` + `socket.gethostname()` |

#### Constraints

- `format_version` must match `^\d+\.\d+$`
- `kdf.memory_cost` minimum: `8192` (8 MiB) — current runtime-enforced lower bound
- `kdf.time_cost` minimum: `1`
- `kdf.parallelism` minimum: `1`
- `salt` must decode to exactly 32 bytes
- `hmac` must decode to exactly 32 bytes
- Timestamps must include timezone information

### 1.3 The `secrets` Object

An alias-keyed secrets pool. Each entry uses a human-readable alias as the map key and also carries an immutable `id` for stable internal references. Multiple aliases may map to the same `key` (environment variable name), enabling management of the same variable across different environments or versions.

```json
{
  "secrets": {
    "prod-db": {
      "id": "sec_4f6f8e9a1c2d3e4f5a6b7c8d9e0f1122",
      "key": "DATABASE_URL",
      "value": "ENC[AES256_GCM,data:cG9zdGdyZXM=,iv:YWJjZGVmZ2hpamts,tag:bW5vcHFyc3R1dnd4,type:str]",
      "description": "Production PostgreSQL",
      "tags": ["production", "database"],
      "created_at": "2026-03-27T10:00:00Z",
      "updated_at": "2026-03-27T10:00:00Z"
    },
    "jwt-secret": {
      "key": "JWT_SECRET",
      "value": "ENC[AES256_GCM,data:and0X3NlY3JldA==,iv:cXdlcnR5dWlvcGFz,tag:enhjdmJubWFzZGZn,type:str]",
      "description": "JWT signing key",
      "tags": ["auth"],
      "created_at": "2026-03-26T14:00:00Z",
      "updated_at": "2026-03-26T14:00:00Z"
    }
  }
}
```

#### Secret Entry Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | `string` | Yes | Immutable secret identifier used for rename-safe references inside app bindings and override maps |
| `key` | `string` | Yes | Environment variable name used for injection. UPPER_SNAKE_CASE recommended. May duplicate across aliases |
| `value` | `string` | Yes | `ENC[…]`-encoded encrypted value |
| `description` | `string` | No | Human-readable description (default: `""`) |
| `tags` | `string[]` | No | Categorization tags (default: `[]`) |
| `created_at` | `string` | Yes | Creation timestamp in ISO 8601 format |
| `updated_at` | `string` | Yes | Last modification timestamp in ISO 8601 format |

#### Alias Naming Rules

| Rule | Description |
|------|-------------|
| Regex | `^[a-z0-9][a-z0-9_-]{0,62}$` |
| Character set | Lowercase letters, digits, hyphens, and underscores |
| First character | Must be a lowercase letter or digit |
| Length | 1–63 characters |
| Uniqueness | Aliases must be unique within the vault |
| Examples | `prod-db`, `staging_redis`, `jwt-secret-v2` |

#### `key` Naming Rules

| Rule | Description |
|------|-------------|
| Character set | `[A-Za-z_][A-Za-z0-9_]*` — valid environment variable names |
| Case | Original case is preserved; all-uppercase is recommended (POSIX convention) |
| Length | Maximum 256 characters |
| Uniqueness | May duplicate across aliases (the same env var name can exist under multiple aliases) |

#### Value Format

All `value` fields must match the `ENC[…]` encoding format (see [§2](#2-encrypted-value-encoding)). Plaintext values and empty strings are not allowed.

### 1.4 The `apps` Object

Application reference configuration. Each application references secrets from the pool via an alias list; at runtime, the corresponding environment variables are injected based on the reference list. Applications do not own encrypted values directly — they hold only alias references and optional injection name overrides.

```json
{
  "apps": {
    "backend": {
      "description": "Backend API service",
      "secrets": ["prod-db", "jwt-secret", "shared-sentry"],
      "overrides": {}
    },
    "frontend": {
      "description": "Frontend build",
      "secrets": ["shared-sentry"],
      "overrides": {
        "shared-sentry": "NEXT_PUBLIC_SENTRY_DSN"
      }
    }
  }
}
```

#### App Entry Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `description` | `string` | No | Human-readable app description |
| `secrets` | `string[]` | Yes | List of alias references from the secrets pool; each element must be an existing alias |
| `overrides` | `object` | No | Map of alias → custom env var name for injection override |

#### Application Name Rules

| Rule | Description |
|------|-------------|
| Character set | `[a-z][a-z0-9_-]*` — must start with a lowercase letter; lowercase letters, digits, underscores, and hyphens are allowed |
| Length | 1–64 characters |
| Reserved names | `_meta` and `secrets` cannot be used as application names |

#### Injection Resolution Algorithm

When running `enva <app_name> -- <command>`, environment variables are resolved and injected as follows:

```python
for alias in apps[app_name]["secrets"]:
    secret = secrets[alias]
    env_name = apps[app_name]["overrides"].get(alias, secret["key"])
    os.environ[env_name] = decrypt(secret["value"])
```

- If the `overrides` map contains the alias, the override value is used as the environment variable name
- Otherwise, `secret["key"]` is used as the environment variable name
- Later entries in the list overwrite earlier ones with the same env var name (list order determines precedence)

## 2. Encrypted Value Encoding

### 2.1 Encoding Format

Each encrypted value is encoded as a self-describing, single-line string inspired by the SOPS ENC format:

```
ENC[AES256_GCM,data:<BASE64>,iv:<BASE64>,tag:<BASE64>,type:<TYPE>]
```

#### Field Descriptions

| Field | Contents | Encoding | Length |
|-------|----------|----------|--------|
| `AES256_GCM` | Algorithm identifier (fixed) | Plaintext | — |
| `data` | Ciphertext (AES-256-GCM encryption output) | Standard Base64 (with `=` padding) | Variable — equal to plaintext length |
| `iv` | Initialization vector / nonce | Standard Base64 (with `=` padding) | Fixed 12 bytes (16 characters after encoding) |
| `tag` | GCM authentication tag | Standard Base64 (with `=` padding) | Fixed 16 bytes (24 characters after encoding) |
| `type` | Original value type identifier | Plaintext | See table below |

#### Delimiter Rules

- Fields are separated by `,` (comma) with **no spaces**
- Field names and values are separated by `:` (colon) with **no spaces**
- The entire expression is wrapped by `ENC[` and `]`

### 2.2 AES-256-GCM Parameters

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Key length | 256 bit (32 bytes) | NIST SP 800-38D |
| Nonce length | 96 bit (12 bytes) | GCM standard recommended length; avoids birthday-bound degradation |
| Authentication tag length | 128 bit (16 bytes) | Maximum GCM tag length for full authentication strength |
| Additional Authenticated Data (AAD) | Key path (see §2.5) | Prevents swapping encrypted values between different aliases |

**Each value uses an independently random nonce**: every encryption operation calls `os.urandom(12)` to generate a fresh nonce. Nonce reuse under the same key is forbidden (a core GCM security requirement).

### 2.3 Supported Types

Before encryption, original values are encoded to byte sequences according to their type; after decryption, they are restored to native Python types using the `type` field.

| type Identifier | Python Type | Encoding to Bytes | Decoding from Bytes | Example Original Value |
|-----------------|------------|-------------------|---------------------|----------------------|
| `str` | `str` | `value.encode('utf-8')` | `bytes.decode('utf-8')` | `"postgresql://..."` |
| `int` | `int` | `str(value).encode('utf-8')` | `int(bytes.decode('utf-8'))` | `5432` |
| `float` | `float` | `str(value).encode('utf-8')` | `float(bytes.decode('utf-8'))` | `0.95` |
| `bool` | `bool` | `b"true"` or `b"false"` | `bytes == b"true"` | `True` |
| `bytes` | `bytes` | Raw bytes | Return directly | `b"\x00\x01\x02"` |

> **Design rationale**: `int` / `float` / `bool` use text encoding rather than binary so that decrypted results remain human-readable during debugging. The `bytes` type is intended for binary data such as certificates and private keys.

### 2.4 Base64 Encoding Convention

- Uses the RFC 4648 §4 standard Base64 alphabet (`A-Z`, `a-z`, `0-9`, `+`, `/`)
- **Padding (`=`) is required** (the default behavior of `base64.b64encode`)
- URL-safe variants are not used (no `-` or `_`)
- No line breaks are inserted

### 2.5 Additional Authenticated Data (AAD)

During AES-256-GCM encryption, the key path is bound to the ciphertext as AAD (Additional Authenticated Data), preventing an attacker from relocating an encrypted value from one alias to another:

```
AAD = "secrets:<alias>"
```

| Scope | AAD Format | Example |
|-------|------------|---------|
| `secrets` | `"secrets:<ALIAS>"` | `"secrets:prod-db"` |

The AAD is passed to the GCM encryption function as UTF-8-encoded bytes. This design is inspired by SOPS's tree-path AAD mechanism (the `additionalData` parameter in `aes/cipher.go`).

### 2.6 Encoding Examples

#### Example 1: String Value

Original value: `"postgresql://user:pass@localhost:5432/mydb"`

```
ENC[AES256_GCM,data:mK3fR7bQ9xYpLdNwA2sTvHjE6iUoZcGk1gDhCqWfXeJm0y4B5nI8rOaS,iv:f0E2gH4iJ6kL8mN0,tag:pQ2rS4tU6vW8xY0zA2bC4d==,type:str]
```

Breakdown:
- `data:mK3fR7…` — the plaintext, UTF-8-encoded then AES-256-GCM-encrypted, then Base64-encoded
- `iv:f0E2gH4iJ6kL8mN0` — Base64 of the 12-byte random nonce (16 characters)
- `tag:pQ2rS4tU6vW8xY0zA2bC4d==` — Base64 of the 16-byte GCM authentication tag (24 characters)
- `type:str` — after decryption, restore as Python `str`

#### Example 2: Integer Value

Original value: `5432`

```
ENC[AES256_GCM,data:NjQzMg==,iv:aB1cD2eF3gH4iJ5k,tag:lM6nO7pQ8rS9tU0vW1xY2z==,type:int]
```

- `data:NjQzMg==` — `"5432"` UTF-8-encoded, encrypted, then Base64-encoded
- `type:int` — after decryption, call `int()` to restore

#### Example 3: Boolean Value

Original value: `True`

```
ENC[AES256_GCM,data:dHJ1ZQ==,iv:zA0yB1xC2wD3vE4u,tag:fG5hH6iI7jJ8kK9lL0mM1n==,type:bool]
```

- `data:dHJ1ZQ==` — `b"true"` encrypted then Base64-encoded
- `type:bool` — after decryption, compare with `b"true"`

#### Example 4: Float Value

Original value: `0.95`

```
ENC[AES256_GCM,data:MC45NQ==,iv:nO2pP3qQ4rR5sS6t,tag:uU7vV8wW9xX0yY1zZ2aA3b==,type:float]
```

#### Example 5: Binary Value

Original value: `b"\x30\x82\x01\x22"` (DER certificate fragment)

```
ENC[AES256_GCM,data:MIIBIQ==,iv:cC4dD5eE6fF7gG8h,tag:iI9jJ0kK1lL2mM3nN4oO5p==,type:bytes]
```

- `type:bytes` — after decryption, return raw bytes without text decoding

### 2.7 Parsing Regular Expression

Implementations may use the following regex to extract each field:

```python
import re

ENC_PATTERN = re.compile(
    r'^ENC\[AES256_GCM,'
    r'data:(?P<data>[A-Za-z0-9+/]+=*),'
    r'iv:(?P<iv>[A-Za-z0-9+/]+=*),'
    r'tag:(?P<tag>[A-Za-z0-9+/]+=*),'
    r'type:(?P<type>str|int|float|bool|bytes)\]$'
)
```

A parse failure (no match) should be treated as vault file corruption, and processing must be refused.

## 3. Key Derivation

### 3.1 KDF Choice: Argon2id

Argon2id (recommended by RFC 9106) is chosen as the password derivation function:

| Property | Description |
|----------|-------------|
| GPU/ASIC resistance | Memory-hard; heavy memory access makes parallel hardware attacks prohibitively expensive |
| Hybrid mode | Argon2id = Argon2i (first pass, side-channel resistant) + Argon2d (subsequent passes, GPU resistant) |
| Standardization | RFC 9106, winner of the Password Hashing Competition |
| Tunable parameters | Three-dimensional tuning of memory, time, and parallelism to suit different hardware |

> **Relationship with the existing Rust HKDF**: The HKDF-SHA256 in the Rust crate is used to derive sub-keys from high-entropy key material (non-password scenarios). The two serve different purposes and coexist independently. See [secrets_manager_tech_decision.md](./secrets_manager_tech_decision.md) §2.

> **Implementation note**: Argon2id is implemented in the Rust `argon2` crate within the `enva-core` library. The vault file format is implementation-agnostic — any compliant reader can decrypt a vault file given the correct password.

### 3.2 Recommended Parameters

| Parameter | Recommended | Minimum | Description |
|-----------|-------------|---------|-------------|
| `memory_cost` | `65536` (64 MiB) | `16384` (16 MiB) | Memory cost in KiB. Higher is more secure but slower |
| `time_cost` | `3` | `1` | Iteration count. Increasing linearly increases compute time |
| `parallelism` | `4` | `1` | Number of parallel lanes. Should not exceed the CPU core count |

**Rationale for parameter choices**:

- `memory_cost=65536` (64 MiB): The OWASP 2024 recommended minimum is 19 MiB (`m=19456, t=2, p=1`). 64 MiB provides a significant security margin while adding only ~100–200 ms of latency on modern development machines.
- `time_cost=3`: Three iterations at 64 MiB typically take 200–400 ms — acceptable for interactive use.
- `parallelism=4`: Matches mainstream development machines (4+ cores); fully utilizes multiple cores without excessive resource consumption.

**Adjustment scenarios**:

| Scenario | Suggested Adjustment |
|----------|---------------------|
| CI/CD environments (resource-constrained) | `memory_cost=32768, time_cost=2` |
| High-security servers | `memory_cost=131072, time_cost=4` |
| Embedded / low-memory devices | `memory_cost=16384, time_cost=5` (trade time for memory) |

### 3.3 Salt Generation

- Length: **32 bytes** (256 bit)
- Source: `os.urandom(32)` (OS CSPRNG)
- Encoding: standard Base64 stored in `_meta.salt`
- Lifetime: **one salt per vault file**, generated at creation time, immutable
- Replacing the salt is equivalent to creating a new vault (all values must be re-encrypted)

```python
import os
import base64

salt = os.urandom(32)
salt_b64 = base64.b64encode(salt).decode('ascii')
# Example output: "x7Kj2mN+pQ8rS4tU6vW8xY0zA2bC4dEfGhIjKlMnOp0="
```

### 3.4 Key Derivation Process

Two independent keys are derived from the user password: one for AES-256-GCM encryption/decryption, and one for HMAC-SHA256 integrity verification.

```
password (user input)
    │
    ├── Argon2id(password, salt, params) → 64 bytes of master key material
    │       │
    │       ├── [0:32]  → encryption_key (256 bit, AES-256-GCM)
    │       │
    │       └── [32:64] → hmac_key (256 bit, HMAC-SHA256)
    │
    (salt read from _meta.salt)
```

**Implementation pseudocode**:

```python
from argon2.low_level import hash_secret_raw, Type

def derive_keys(
    password: str,
    salt: bytes,
    memory_cost: int,
    time_cost: int,
    parallelism: int,
) -> tuple[bytes, bytes]:
    """Derive encryption and HMAC keys. Returns (encryption_key, hmac_key)."""
    raw = hash_secret_raw(
        secret=password.encode('utf-8'),
        salt=salt,
        time_cost=time_cost,
        memory_cost=memory_cost,
        parallelism=parallelism,
        hash_len=64,           # 64-byte output
        type=Type.ID,          # Argon2id
    )
    encryption_key = raw[:32]  # first 32 bytes
    hmac_key = raw[32:]        # last 32 bytes
    return encryption_key, hmac_key
```

**Why derive 64 bytes in one call instead of two 32-byte calls**:

A single Argon2id invocation outputs 64 bytes, which is then split into two 32-byte keys. Compared to calling Argon2id twice with different salts or contexts:
1. **Performance**: Only one memory-hard computation (~200–400 ms) instead of two
2. **Security**: Argon2id output is already a cryptographically secure pseudo-random sequence; the first and last 32 bytes are computationally unlinkable
3. **Simplicity**: Reduces parameter management complexity

### 3.5 Password Strength Requirements

Vault implementations should enforce (can be disabled via configuration):

| Check | Rule |
|-------|------|
| Minimum length | 12 characters |
| Character diversity | Must include at least two of: uppercase, lowercase, digits |
| Entropy estimate | Recommended ≥ 60 bits (may be evaluated with libraries such as `zxcvbn`) |

Password strength checks are performed at the CLI layer; the vault format itself does not record any password information.

## 4. HMAC Integrity

### 4.1 Purpose

The HMAC check covers the data integrity of the vault file, defending against the following attacks:

| Attack Type | Defense Mechanism |
|-------------|-------------------|
| Tampering with encrypted values | HMAC covers all `ENC[…]` strings |
| Deleting or adding aliases/keys | HMAC covers all alias names and key values |
| Reordering entries | Canonical serialization eliminates ordering ambiguity |
| Tampering with app references | HMAC covers all app names and secrets reference lists |
| Downgrading KDF parameters | HMAC covers `format_version` and `kdf` |
| Replacing the entire vault (different salt) | The HMAC key is derived with the salt; a different salt yields a different HMAC key |

> AES-GCM authentication tags only protect the integrity of individual values. The HMAC provides file-level global integrity, ensuring that the combination of all values has not been tampered with.

### 4.2 Algorithm

**HMAC-SHA256**, keyed with the `hmac_key` derived in §3.4 (32 bytes).

### 4.3 HMAC Key

The HMAC uses a 256-bit key **independent of** the encryption key, provided by the last 32 bytes of the same Argon2id derivation (see §3.4). Key separation ensures:
- Leaking the encryption key does not compromise HMAC verification capability
- Leaking the HMAC key does not compromise ciphertext security

### 4.4 Canonical Serialization

The HMAC input is a canonical serialization of the vault data as a byte sequence, constructed as follows:

```
HMAC_INPUT = CANONICAL_SERIALIZE(_meta, secrets, apps)
```

#### HMAC Coverage

- All alias names (sorted)
- All `secret.key` values (sorted by alias)
- All `secret.value` values (sorted by alias)
- All app names (sorted)
- All `app.secrets` reference lists (sorted by app name)
- `_meta.format_version` + `_meta.kdf`

#### Serialization Steps

**Step 1**: Serialize the protected fields from `_meta`

```
meta_part = "format_version:" + _meta.format_version + "\n"
          + "kdf.algorithm:" + _meta.kdf.algorithm + "\n"
          + "kdf.memory_cost:" + str(_meta.kdf.memory_cost) + "\n"
          + "kdf.time_cost:" + str(_meta.kdf.time_cost) + "\n"
          + "kdf.parallelism:" + str(_meta.kdf.parallelism) + "\n"
```

**Step 2**: Collect entries from the `secrets` pool

For each alias in `secrets` (sorted by alias in UTF-8 byte order):
```
"secrets:<ALIAS>:key=" + secret.key
"secrets:<ALIAS>:value=" + secret.value
```

**Step 3**: Collect entries from `apps`

For each app in `apps` (sorted by app name in UTF-8 byte order):
```
"apps:<APP>:secrets=" + ",".join(app.secrets)
```

**Step 4**: Sort all entries in ascending UTF-8 byte order (ensures deterministic ordering)

**Step 5**: Concatenate into the final input

```
HMAC_INPUT = meta_part + sorted_entries.join("\n") + "\n"
```

#### Example

Given the following vault contents:

```json
{
  "_meta": {
    "format_version": "2.0",
    "kdf": { "algorithm": "argon2id", "memory_cost": 65536, "time_cost": 3, "parallelism": 4 },
    "salt": "...", "hmac": "...",
    "created_at": "...", "updated_at": "...", "created_by": "..."
  },
  "secrets": {
    "prod-db": {
      "key": "DATABASE_URL",
      "value": "ENC[...db...]",
      "description": "Production DB",
      "tags": ["production"],
      "created_at": "...", "updated_at": "..."
    },
    "jwt-secret": {
      "key": "JWT_SECRET",
      "value": "ENC[...jwt...]",
      "description": "JWT key",
      "tags": ["auth"],
      "created_at": "...", "updated_at": "..."
    }
  },
  "apps": {
    "backend": {
      "description": "Backend API",
      "secrets": ["prod-db", "jwt-secret"],
      "overrides": {}
    }
  }
}
```

The canonical serialization result (sorted by UTF-8 byte order):

```
format_version:2.0
kdf.algorithm:argon2id
kdf.memory_cost:65536
kdf.time_cost:3
kdf.parallelism:4
apps:backend:secrets=prod-db,jwt-secret
secrets:jwt-secret:key=JWT_SECRET
secrets:jwt-secret:value=ENC[...jwt...]
secrets:prod-db:key=DATABASE_URL
secrets:prod-db:value=ENC[...db...]
```

> Note: `apps:backend:secrets…` < `secrets:jwt-secret:key…` (in UTF-8 byte order). `description`, `tags`, `created_at`, `updated_at` and other metadata fields are excluded from the HMAC. `_meta.salt`, `_meta.hmac`, and timestamps are also excluded — the salt is implicitly bound via key derivation, and the HMAC is an output, not an input.

### 4.5 HMAC Computation

```python
import hmac
import hashlib

def compute_hmac(hmac_key: bytes, canonical_input: bytes) -> bytes:
    """Compute HMAC-SHA256."""
    return hmac.new(hmac_key, canonical_input, hashlib.sha256).digest()
```

The result is 32 bytes, standard-Base64-encoded before being stored in `_meta.hmac`.

### 4.6 Verification Timing and Flow

#### Verification on Load (mandatory for every vault read)

```
1. Read the vault JSON
2. Extract salt and KDF parameters from _meta
3. Prompt the user for the password
4. derive_keys(password, salt, kdf_params) → (encryption_key, hmac_key)
5. Canonically serialize the vault data → canonical_input
6. Compute expected_hmac = HMAC-SHA256(hmac_key, canonical_input)
7. Read _meta.hmac → stored_hmac
8. Perform constant-time comparison using hmac.compare_digest(expected_hmac, stored_hmac)
9. If mismatch → refuse decryption, raise error "vault integrity check failed"
```

> **Constant-time comparison**: `hmac.compare_digest()` must be used instead of `==` to prevent timing side-channel attacks.

#### Verification on Save (mandatory for every vault write)

```
1. Encrypt all new/modified values
2. Canonically serialize all data → canonical_input
3. Compute new_hmac = HMAC-SHA256(hmac_key, canonical_input)
4. Base64-encode new_hmac and write to _meta.hmac
5. Update _meta.updated_at
6. Atomically write the vault file (write-to-temp + rename)
```

#### Error Handling

| Verification Result | Action |
|---------------------|--------|
| HMAC matches | Proceed with normal decryption |
| HMAC mismatch (correct password) | File has been tampered with; refuse operation; log to audit trail |
| HMAC mismatch (wrong password) | Prompt "incorrect password or corrupted vault" |
| `_meta.hmac` missing | Treat as format error; refuse operation |

> The external error message does not distinguish between "wrong password" and "file tampered" to avoid information leakage. Internally (audit log), the specific cause is recorded.

## 5. Version Evolution Strategy

### 5.1 Version Number Semantics

`format_version` follows the `major.minor` format:

| Change Type | Version Change | Meaning |
|-------------|----------------|---------|
| Add optional field | minor +1 | Forward-compatible; older readers ignore unknown fields |
| Add required field | major +1 | Breaking change; older readers cannot parse |
| Change field semantics | major +1 | Breaking change |
| Change encryption algorithm | major +1 | Breaking change |
| Change HMAC coverage scope | major +1 | Breaking change |
| Add new type option (e.g. `json`) | minor +1 | Older readers may fall back to `str` for unknown types |

### 5.2 Compatibility Rules

#### Read Compatibility Matrix

| Reader Version | File format_version | Behavior |
|----------------|---------------------|----------|
| v1.x | `1.0` | Fully supported |
| v1.x | `2.0` | **Rejected** — prompts user to upgrade |
| v2.x | `1.0` | Supported (backward-compatible read + optional migration) |
| v2.x | `2.0` | Fully supported |
| v2.x | `2.1` | Supported (unknown fields ignored) |
| v2.x | `3.0` | **Rejected** — prompts user to upgrade |

#### Read Rules

```python
def check_version_compatibility(file_version: str, reader_version: str) -> None:
    """Check compatibility between file version and reader version."""
    file_major = int(file_version.split('.')[0])
    reader_major = int(reader_version.split('.')[0])

    if file_major > reader_major:
        raise IncompatibleVersionError(
            f"Vault format v{file_version} requires reader v{file_major}.x+, "
            f"current reader is v{reader_version}. Please upgrade."
        )
    # file_major <= reader_major: readable (backward compatible)
```

#### Unknown Field Handling

- **On read**: Ignore unknown fields in `_meta` (forward compatibility)
- **On write**: Preserve unknown fields from the original file (round-trip fidelity)
- **HMAC computation**: Only covers the canonical fields defined in §4.4; unknown fields are excluded from the HMAC

### 5.3 Version Evolution Path

#### v1.0 (Historical Version)

The initial version:
- Argon2id KDF
- AES-256-GCM per-value encryption
- HMAC-SHA256 integrity
- `global` + `apps` two-level structure (flat key-value)
- Supported types: `str`, `int`, `float`, `bool`, `bytes`

#### v2.0 (Current Version)

Breaking upgrade — introduces the alias-based data model:
- `global` removed, replaced by the `secrets` alias pool
- `apps` changed from value-ownership model to reference model
- Each secret entry contains `key`, `value`, `description`, `tags`, `created_at`, `updated_at`
- Applications reference secrets via alias lists, with `overrides` for injection name customization
- HMAC coverage updated: covers aliases, keys, values, and app reference lists

#### v2.1 (Planned) — New Optional Fields

A forward-compatible extension:

```json
{
  "_meta": {
    "format_version": "2.1",
    "kdf": { "…": "…" },
    "salt": "…",
    "hmac": "…",
    "created_at": "…",
    "updated_at": "…",
    "created_by": "…",
    "description": "Production vault for project X",
    "last_rotated_at": "2026-06-01T00:00:00+08:00",
    "rotation_policy_days": 90
  }
}
```

New optional fields:
- `_meta.description`: Description of the vault's purpose
- `_meta.last_rotated_at`: Last password rotation timestamp
- `_meta.rotation_policy_days`: Password rotation policy in days

v2.0 readers encountering these fields will ignore them and function normally.

#### v2.2 (Planned) — New Type

Adds `type:json` to support encrypted structured data:

```
ENC[AES256_GCM,data:…,iv:…,tag:…,type:json]
```

v2.0/v2.1 readers encountering `type:json` may fall back to `type:str` and return a JSON string.

#### v3.0 (Future) — Breaking Changes

Possible scenarios requiring breaking changes:

| Change | Reason |
|--------|--------|
| Replace KDF algorithm | If a security vulnerability is found in Argon2id, migration to a successor is needed |
| Modify HMAC coverage scope | Extend HMAC coverage to new required fields |
| Change ENC encoding format | Support new encryption algorithm identifiers (e.g. `XCHACHA20_POLY1305`) |

### 5.4 Migration Mechanism

#### Automatic Migration (minor versions)

Minor version upgrades require no migration. Readers ignore unknown fields; writers automatically add default values for new fields on save.

#### Manual Migration (major versions)

```
enva vault migrate --vault vault.json --target-version 2.0
```

Migration flow:

```
1. Read the old-version vault (using old-version read logic)
2. Decrypt all values
3. Create a new-version vault structure
4. Re-encrypt all values with new parameters
5. Compute the new HMAC
6. Write the new file (or --in-place to overwrite)
7. Verify that the new file can be read successfully
```

**v1.0 → v2.0 migration mapping**:
- Each `KEY: ENC[…]` in `global` → an entry in the `secrets` pool with an auto-generated alias (lowercased key)
- Each `APP.KEY: ENC[…]` in `apps` → an entry in the `secrets` pool aliased as `<app>-<lowercase_key>`, with the app's reference list populated automatically

**Security requirements**:
- During migration, plaintext exists only in memory
- The original file is automatically backed up before migration (`vault.json.bak.TIMESTAMP`)
- The new vault's integrity is verified after migration

## 7. Security Considerations

### 7.1 Threat Model and Defenses

| Threat | Attack Vector | Defense Measure |
|--------|---------------|-----------------|
| Offline brute force | Attacker obtains vault file and tries passwords offline | Argon2id memory-hard KDF; 64 MiB + 3 iterations costs 200–400 ms per attempt |
| Ciphertext swapping | Attacker moves alias A's ciphertext to alias B | AAD binds the alias path (§2.5); GCM authentication fails |
| File tampering | Attacker modifies any field in the vault file | HMAC-SHA256 global integrity check (§4) |
| App reference tampering | Attacker modifies the secrets reference list in apps | HMAC covers app reference lists |
| KDF parameter downgrade | Attacker reduces memory_cost to the minimum | HMAC covers KDF parameters; tampering causes HMAC verification failure |
| Nonce reuse | Implementation bug causes same nonce to encrypt different values | Each encryption call generates an independent nonce via `os.urandom(12)` |
| Memory residue | Plaintext remains in process memory after decryption | Proactive zeroing on completion (`SecureString` pattern); use `ctypes.memset` in Python |
| Timing side-channel | Infer HMAC value through comparison timing | Constant-time comparison via `hmac.compare_digest()` |

### 7.2 Out-of-Scope Scenarios

| Scenario | Reason |
|----------|--------|
| Memory dump attacks (cold boot / swap) | Requires OS-level protection (mlock); beyond application-layer scope |
| Malicious reader | A reader can access plaintext; cannot be defended at the format level |
| Password leakage (phishing / keylogging) | Beyond the scope of the file format |
| Quantum computing attacks | AES-256 retains 128-bit security under quantum scenarios (Grover); no migration needed yet |

### 7.3 Implementation Checklist

Implementers should ensure the following:

- [ ] Each encryption uses `os.urandom(12)` to generate an independent nonce
- [ ] GCM authentication tag length is fixed at 128 bits
- [ ] The password is never persisted in any form (not written to files, not logged)
- [ ] HMAC comparison uses `hmac.compare_digest()`
- [ ] File writes use atomic write (`write-to-temp` + `os.rename`)
- [ ] HMAC is verified before decryption on load
- [ ] Unknown `format_version` major versions are rejected
- [ ] `_meta.kdf.memory_cost` minimum value check (`>= 16384`)
- [ ] Error messages do not distinguish "wrong password" from "corrupted file" (unified external message)
- [ ] Alias references in app `secrets` lists are validated for existence at parse time
